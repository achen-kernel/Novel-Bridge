# 苍穹外卖（Sky Take-Out）项目结构说明文档

## 一、项目概览

苍穹外卖是一个面向平台管理员、商家端和 C 端消费者的餐饮外卖交易平台，采用 **Maven 多模块聚合工程** 架构。

| 项目信息 | 说明 |
|----------|------|
| 父项目 | `sky-take-out`（`pom.xml`，聚合三个子模块） |
| Spring Boot | 2.7.3 |
| MyBatis-Plus | 3.5.3.1 |
| 数据库 | MySQL + Redis |
| 消息队列 | RabbitMQ |
| 接口文档 | Knife4j (Swagger) |
| 构建工具 | Maven |

└─────────────────────────────────────────────────────────────────┘

### 模块依赖关系表

| 模块 | 依赖关系 | 说明 |
|------|----------|------|
| **sky-common** | 无内部模块依赖 | 基础公共模块，最底层，不依赖其他业务模块 |
| **sky-pojo** | 无内部模块依赖 | 数据模型模块，独立定义所有 DTO/Entity/VO |
| **sky-server** | 依赖 sky-common + sky-pojo | 业务主模块，引入两个基础模块构建完整应用 |

> **要点**：`sky-common` 与 `sky-pojo` 之间没有依赖关系，它们是平级的底层基础模块。`sky-server` 同时依赖两者，三者构成经典的两层依赖结构。

### 3.3 核心设计规范

1. **统一响应模型**：所有 Controller 返回 `Result<T>`，code=1 为成功，code=0 为失败，前端可统一处理。
2. **异常体系**：继承 `BaseException(RuntimeException)`，在 `GlobalExceptionHandler` 中统一捕获处理。
3. **线程上下文**：`BaseContext` 使用 `ThreadLocal` 管理当前请求的用户ID、店铺ID、角色编码，配合拦截器在请求入口设置、在请求完成时清理。
4. **工具类静态方法**：`JwtUtil`、`AliOssUtil` 等为无状态工具类，可直接调用。
5. **配置属性集中管理**：将 application.yml 中的外部配置（OSS、JWT、微信）映射为 `@ConfigurationProperties` 类。

### 4.3 核心设计规范

1. **三模型分离**：
   - **DTO**：仅包含前端传入的必要字段，不包含敏感内部字段（如密码）。如 `EmployeeDTO` 不包含 `password`。
   - **Entity**：完整映射数据库表字段，包含创建/更新时间等审计字段。使用 `@Data` + `@Builder` + `NoArgsConstructor` + `AllArgsConstructor`。
   - **VO**：可继承 Entity（如 `OrderVO extends Orders`），也可独立定义，按前端展示需求组合数据。

2. **ORM 注解策略**：
   - 主键使用 `@TableId(type = IdType.ASSIGN_ID)`，采用雪花算法生成全局唯一 ID。
   - 无 `@TableName` 注解，表名默认使用类名的下划线命名（如 `Orders` → `orders` 表）。
   - `Orders` 实体中的 `version` 字段配合乐观锁插件使用。

3. **Lombok 标配**：所有 POJO 类标配 `@Data`，部分使用 `@Builder` 链式构造。

4. **序列化**：实现 `Serializable` 接口，保证对象可在网络传输和缓存中序列化。

5. **订单状态常量**：在 `Orders` 实体内部定义状态常量（如 `PENDING_PAYMENT=1`、`COMPLETED=5` 等），集中管理业务状态值。

6. **防重复提交**：`OrdersSubmitDTO` 包含 `orderToken` 字段，配合 Redis + Lua 实现请求幂等性。

### 5.3 核心设计规范

#### 5.3.1 分层架构

```
HTTP Request
    │
    ▼
┌─────────────────────┐
│  Controller 层       │  接收请求、参数校验、调用Service、返回Result
│  (@RestController)   │  按端划分：admin / merchant / user / notify
├─────────────────────┤
│  Interceptor 层      │  JWT令牌校验、用户身份提取、ThreadLocal上下文设置
│  (HandlerInterceptor)│  三套拦截器分别拦截三端请求
├─────────────────────┤
│  Service 层          │  业务逻辑编排、事务管理、调用Mapper
│  (接口 + Impl)       │  接口定义契约、Impl实现具体逻辑
├─────────────────────┤
│  Mapper 层           │  数据库访问、SQL映射（注解+XML两种方式）
│  (@Mapper)           │  支持 @AutoFill 注解实现自动填充
├─────────────────────┤
│  MyBatis-Plus        │  分页插件、乐观锁插件、自动填充处理器
└─────────────────────┘
    │
    ▼
  Database (MySQL + Redis)
```

#### 5.3.2 Controller 层分组策略

根据 **端（Client）** 划分，而非按功能模块划分：

| 分组 | 路由前缀 | 使用方 | 鉴权方式 |
|------|----------|--------|----------|
| `controller/admin/` | `/admin/**` | 平台管理员 | JwtTokenAdminInterceptor |
| `controller/merchant/` | `/merchant/**` | 商家 | JwtTokenMerchantInterceptor |
| `controller/user/` | `/user/**` | C端消费者 | JwtTokenUserInterceptor |
| `controller/notify/` | 外部回调 | 微信支付 | 无拦截器 |

#### 5.3.3 AOP 架构亮点

**公共字段自动填充机制**：
```
@AutoFill(OperationType.INSERT)   ← 标注在Mapper方法上
        │
        ▼
AutoFillAspect (AOP切面)          ← 前置通知
        │
        ├─ INSERT操作: 反射调用 setCreateTime / setCreateUser / setUpdateTime / setUpdateUser
        └─ UPDATE操作: 反射调用 setUpdateTime / setUpdateUser
```

**权限校验机制**：
```
@RequiresPermission("permission_code")   ← 标注在Controller方法上
        │
        ▼
PermissionAspect (AOP切面)               ← 前置通知
        │
        ├─ 平台管理员 (PLATFORM_ADMIN) → 直接放行
        └─ 其他角色 → 查询角色权限列表 → 判断是否包含所需权限
```

## 七、架构设计评价

### 7.1 优势

1. **模块职责清晰**：`common` 纯基础设施、`pojo` 纯数据模型、`server` 纯业务逻辑，各司其职，无循环依赖。
2. **统一响应与异常**：`Result<T>` + `BaseException` + `GlobalExceptionHandler` 形成完整的异常链路，前端可统一处理错误。
3. **AOP 应用合理**：公共字段填充和权限校验使用 AOP 实现，与业务代码解耦，减少重复代码。
4. **多端接口分组**：Controller 按 admin/merchant/user/notify 划分，配合三套拦截器实现不同端的独立鉴权。
5. **三模型分离**：DTO/Entity/VO 各司其职，避免将数据模型与展示模型混为一谈。
6. **防重复提交**：`orderToken` 机制保证订单创建接口的幂等性。
7. **中间件集成适度**：Redis、RabbitMQ、WebSocket 等中间件的使用定位明确，没有过度设计。

### 7.2 可优化的方面

1. **Entity 的 ORM 注解不统一**：部分实体使用 MyBatis-Plus 的 `@TableId`，大部分未使用。建议统一规范，明确哪些实体需要显式注解。
2. **DTO 数量偏多**：29 个 DTO 中部分功能相似，可考虑通过组合与继承减少重复定义。
3. **Service 层接口与实现**：每个 Service 都有一个接口和一个实现类，对于简单 CRUD 服务来说略显冗余。可考虑对纯数据访问服务简化。
4. **常量分散**：状态值部分定义在 Entity 内部（如 `Orders`），部分可能散落在 Service 中，建议统一管理。

## 八、总结

苍穹外卖项目采用 **经典的单体分层架构**，以 Maven 多模块组织代码，各模块分工明确：

- **sky-common** 构建了基础设施层（工具/常量/异常/响应/配置）
- **sky-pojo** 构建了数据模型层（DTO/Entity/VO）
- **sky-server** 构建了业务逻辑层（Controller/Service/Mapper/AOP/配置）

三层模块形成 `common + pojo ← server` 的依赖关系，结构清晰，易于扩展。在遵循经典 Spring Boot 最佳实践的同时，项目通过 AOP 切面、多端拦截器、防重复提交等机制展现了良好的工程设计水平。
