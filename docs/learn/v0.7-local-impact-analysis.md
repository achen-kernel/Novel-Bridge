# v0.7 对本地端的影响分析

> 基于 `NovelBridge_35B教师数据工程与叙事图谱训练规划_v0.7.md`
> 对比之前使用的 `内部研究版与开源标准版技术路线规划_v0.4.md`

---

## 一、核心转向

v0.7 最关键的变更：

| 维度 | v0.4 | v0.7 |
|---|---|---|
| **首要任务** | 扩展 Demo 功能 | 建立训练数据工程闭环 |
| **API 角色** | 业务编排 + 前端网关 | 仅承担前置概览 + 抽样审查 |
| **新引入** | 轻量候选、规则预处理 | 叙事阶段/故事线建模 |
| **数据库设计** | 20 表一体化设计 | 按层分表（静态/阶段/状态/训练） |
| **核心产出** | 可运行的 Demo 系统 | 高质量 `accepted_train.jsonl` |

关键原文：

> "不应继续优先扩展 Demo 功能，而应先建立稳定的训练数据工程闭环"

这意味着**本地端 Java 不需要大量新增功能**，重点是支持训练数据管线的运行和审查。

## 四、本地端下一步建议

### P0：external_reference API（用户粘贴外部资料）

v0.7 §4 明确说外部资料的第一优先级是用户手动粘贴。当前本地端完全没有这个能力。

建议做一个简单的 API：

```http
POST /api/books/{bookId}/external-references
{
  "sourceType": "USER_PASTE",
  "content": "用户粘贴的百科/书评/梗概...",
  "summary": "可选摘要"
}
```

存储：远端 MySQL `external_reference` 表（待远端创建）。
用途：辅助 35B 生成更准确的 book_overview 和训练样本。

### P1：BookOverview 表结构调整跟踪

v0.7 第 9.1 节把 `book_overview` 从 `novel_book_source` 的一列拆成独立表。
需要等远端确定后，把 Java 端的读写逻辑从 `BookSourceMapper` 迁移到新的 `BookOverviewMapper`。

### P2：训练样本浏览 API（可选）

提供一个只读 API，让前端可以查看 `accepted_train.jsonl` 的内容和统计。
但不是必须的——训练数据工程主要在远端 Python 侧闭环。

## 五、总结

**v0.7 实际上减轻了本地端的开发压力。** 之前 v0.4 规划了大量 Demo 功能（审核面板、卡片系统、跨书比较），现在这些全部让位于训练数据工程闭环，而这些主要在远端 Python + 35B 侧。

本地端只需要：
1. ✅ 保持已有的 Neo4j/Review/Overview 三个模块
2. ➕ 新增 external_reference API（用户粘贴外部资料）
3. ➕ 等远端确定 `book_overview` 独立表结构后调整

其他不动。
